# V8 Introspection — Council-grade proposal

**Filed:** 2026-04-26
**Author:** ASO brain (drafted by Claude Opus 4.7 in conversation with Sahil)
**Status:** Proposal — awaiting Sahil approval before any v8 code lands
**Supersedes:** v7 hand-written `<game>-v6.json` specs (operational until v8 ships)

---

## TL;DR

The current ASO brain is a downstream consumer of hand-written specs. It produces conversion-grade carousels for games whose interesting states are 1-click reachable (like AP) but punts on games whose interesting states require state evolution (like BX) — falling back on "wait for game-repo dev-hooks to ship one by one".

**v8 makes the brain a first-class game introspector.** It reads the game source on first encounter, builds an inventory of exciting state primitives, designs minimal hooks where needed (one PR not five), and drives the live game to every state with crafty Playwright sequences. Self-imposed bottlenecks ("BX is blocked") become rare; bottlenecks that remain become honest ("we tried these three crafty paths, all failed, here's the minimal hook that unblocks everything").

**Why this matters now:** Sahil 2026-04-26 — *"we are not here to do minimum creatives, we are here to do fucking ASO"* and *"I want a smart brain only optimized to think about ASO that can summon smart af agents who build in crafty ways and not create their own bottlenecks and limitations"*.

## Why v7 fails

### Symptom 1 — BX has 3 shots, "compliant" but half-empty

BX captured menu / adventure-map / classic-empty. Combo / mega-combo / best-score-celebration / revive / adventure-themed-mid all ship as "deferred until dev-hooks land" (Bloxplode-Beta #72-#76, five separate game-repo issues). Brain framed this as compliance-acceptable; Sahil rejected the framing.

### Symptom 2 — BX `?screenshot=1` design = 5 issues for what should be 1 PR

The original BX dev-hook design (Ripon's #72-#76) is `?screenshot=1&state=combo&multiplier=N&score=915`, `?screenshot=1&state=adventure&level=2&progress=100`, etc. Each state gets its own URL parser branch, its own state-injector, its own smoke test. Total surface area: ~80-200 lines across 5 PRs. Brain did not propose a smaller alternative.

A smarter alternative the brain could have proposed: `window.__bx_dev = { setState(json) {...} }` — single 5-line export, brain calls it via `page.evaluate()` with the desired state JSON. ONE PR. Same 5 carousels reachable. **The brain did not introspect BX source to discover that the existing combo/score/best systems already live in JavaScript objects that just need a setter exposed.**

### Symptom 3 — AP v7 hero shot is a fragile UI-navigation hack

Brain navigates menu → levels grid → click level 53 tile → capture. Works, but:
- Procgen variance: some Level 53 seeds will land sparser than others
- Selector brittleness: the level grid uses `.path-node` (not `.level-cell` as I first assumed)
- Text matching: `.path-node` text content is `53—` not `53`, requires regex extraction
- One game-repo refactor of the level grid component breaks the entire AP carousel silently

A smarter alternative: brain reads AP source, finds `controller.startLevel(N)` is exposed on `window.controller` (or could be), and calls it directly from `page.evaluate()`. Reproducible. Refactor-resistant. No selector dance.

### Symptom 4 — narrow framing missed reachable variants

Even within v7's "wait for dev-hooks" stance, BX has 3-5 more naturally reachable states the brain didn't add (adventure level 1 gameplay, adventure level 5 different theme, settings overlay, pause menu, adventure map scrolled). Brain stopped at "what was already in the spec" because the prompt to expand wasn't strong enough.

### The common root cause

All four symptoms share one cause: **the brain operates as a downstream consumer of hand-written specs, with no model of the games themselves.** It can only do what the spec says, and the spec's quality is bounded by what a human (Sahil, Ripon, or me-the-LLM) typed in. No introspection, no creativity beyond the spec, no model of "what makes this game interesting".

## v8 vision

### Brain as game introspector

For every game in the portfolio, the brain produces and maintains a **Game Inventory** (`brain/aso/inventories/<game>.md`):

```markdown
# Game Inventory — <game name>
# Generated by game-introspector subagent on <ISO date>
# Source SHA: <git SHA at introspection time>

## Exciting state primitives
- combo (BX): triggered when N consecutive line-clears chain. Multiplier shown as `×N`. State lives in `src/classic/combo.js:42` as `comboMultiplier` global.
- best_score (BX): celebration overlay shown post-game when score > stored best. Lives in `src/classic/score.js:88`. Triggered via `showBestScoreModal(score)`.
- mega_combo (BX): cosmetic overlay for multiplier ≥ 8. Lives in `src/classic/effects.js:120`.
- ...

## Already-exposed dev hooks
- `?dev=1` — house-mafia only, enables single-player mock mode (memory: `project_house_mafia_dev_mode.md`)
- `?reset=1` — AP, clears localStorage at boot
- `window.controller` — AP, exposes game controller including `startLevel(N)` (CONFIRMED via grep `src/main.js:200`)

## Minimal hooks brain proposes adding
- BX: `window.__bx_dev = { setState({combo, score, level, theme, ...}) }` — 5 lines, exposed under dev build flag, gates entire ASO carousel
- HM: `window.__hm_dev = { mockSession({playerCount, names, role}) }` — 8 lines, mocks Supabase channel client-side

## Renderer entry points
- AP: `src/rendering/canvas-renderer.js:render()` — paints arrows + grid + HUD
- BX: `src/classic/board-renderer.js:render(state)` — pure function, easy to call with seeded state

## Theme/palette tokens
- AP: `src/styles/themes.css` — `theme-default` (cream + orange), `theme-dark` (gray + amber), `theme-ocean` (mint + teal)
- BX: `src/styles/palette.css` — primary cyan + accent yellow + 5 adventure-level themes

## Game flow graph
- AP boot → if !tutorial_complete → tutorial → menu → {play, levels, daily, settings}
- BX boot → splash (4.5s) → menu → {classic, adventure} → ...

## "What makes this game addictive" (ASO hypothesis)
- AP: "I can solve THAT" — aspirational difficulty + procgen depth + daily streak retention
- BX: "WOAH that combo" — dopamine peaks from chain explosions + mastery curve via multiplier ceiling
- HM: "play with friends right now" — social moments + secret roles + stakes

## Density estimates
- Level 53 procgen: avg 32 arrows, ~38% cell occupancy (above AP density floor of 35%)
- Level 80 procgen: avg 22-41 arrows (variance), some seeds below floor (see arrow-puzzle-testing #218)
- ...
```

This inventory is the brain's **model of the game**. Every ASO decision flows from it.

### Brain as smart-agent coordinator

The brain doesn't do the heavy lifting itself. It coordinates four specialized subagents:

#### 1. `game-introspector`
- **Trigger:** First encounter with a game, or game-repo merges to non-cosmetic source files (`src/game/`, `src/levels/`, `src/screens/`, `src/rendering/`)
- **Tools:** Read, Glob, Grep, Bash (limited to game-repo + git)
- **Input:** game name + source SHA
- **Output:** `brain/aso/inventories/<game>.md` (overwrite if exists)
- **Crafty pattern:** reads `src/main.js`, traces imports, identifies state primitives by their identifiers (`combo`, `score`, `streak`, `level`, `theme`), confirms by reading their setters, classifies by gameplay role

#### 2. `state-reacher`
- **Trigger:** ASO brain plans a carousel and needs a path to each shot's state
- **Tools:** Read, Bash (Playwright invocation only)
- **Input:** target state spec (e.g. `{game: bloxplode, state: classic-combo-x4-score-915}`)
- **Output:** one of three results:
  - (a) **Playwright sequence using existing primitives** — `page.evaluate(() => window.__bx_dev.setState({combo: 4, score: 915}))` + selector wait
  - (b) **Minimal-PR proposal** — "add `window.__bx_dev = { setState(json) }` in `src/dev-hooks.js`, gate by `import.meta.env.DEV`, 8 lines total"
  - (c) **Three-paths-failed report** — what was tried, why each failed, recommended next step
- **Crafty pattern:** ALWAYS tries (a) first using inventory.md. Falls to (b) only when (a) genuinely impossible. Never jumps to (c) without trying (a) and (b).

#### 3. `hook-designer`
- **Trigger:** `state-reacher` returns result type (b) — minimal hook needed
- **Tools:** Read, Edit (game source READ ONLY, output is a description not a code change)
- **Input:** state setters that need exposing + game's existing module conventions
- **Output:** PR description for the game repo with the minimum-viable code change
- **Crafty pattern:**
  - Bias toward `window.__game_dev_*` exports (5-10 lines) over URL routing (50-200 lines)
  - Bias toward ONE consolidated hook per game over per-state hooks
  - Bias toward `Object.assign(window, {__game_dev_setters})` in a single `src/dev-hooks.js` file behind `import.meta.env.DEV` gating
  - Output includes acceptance criteria, smoke test spec, and exact diff if possible

#### 4. `carousel-composer`
- **Trigger:** Inventory + reach-results ready
- **Tools:** Read, Write (output spec only)
- **Input:** inventory.md + per-shot reach-results + per-genre hook-ordering canon + competitor research
- **Output:** generated `<game>-v6.json` with 10 shots in conversion-psychology order
- **Crafty pattern:**
  - Hero shot (#1) is ALWAYS the highest-stopping-power state from inventory's "addictive" hypothesis
  - Captions denylist-checked inline before write
  - Story-flow ordering: hero → variety → retention → depth → personalization → brand
  - All shots provenance-tagged `playwright-capture` (compliance-clean)

### Pipeline becomes generated, not hand-authored

```
v7 (current):                          v8 (target):

human writes <game>-v6.json            game-introspector → inventory.md
       │                                       │
       ▼                                       ▼
capture.mjs reads spec                 carousel-composer → <game>-v6.json (generated)
       │                                       │
       ▼                                       ▼
output PNGs                            capture.mjs reads spec → output PNGs
                                               │
                                               ▼
                                       (cleanup, audit, log, refresh inventory)
```

The human's role shifts from "write specs" to "review carousels + endorse hook proposals". Higher leverage, less typing, no selector debugging.

## What v8 unlocks beyond ASO

The Game Inventory is a brain primitive other agents consume:

- **UA agent** consumes `inventory.md` → "what makes this game addictive" → ad creative briefs + ASO copy
- **Content agent** consumes `inventory.md` → "what loops are exposed" → suggests level / mode / event additions that extend existing addictive loops (not random new mechanics)
- **Monetization agent** consumes `inventory.md` → "where are the dopamine peaks" → suggests ad placement timing (interstitial after combo-x8 not random)
- **Competitor agent** consumes `inventory.md` → makes apples-to-apples comparisons (BX combo system vs Block Blast combo system) instead of vague genre comparisons
- **Council** consumes `inventory.md` → tracks "addictive hypothesis stability" across game iterations as a leading indicator of player retention

The introspection brain is the **precursor to a much smarter factory across every agent**, not just ASO. ASO is the trigger because Sahil's frustration surfaced the gap, but the leverage is portfolio-wide.

## Risks + mitigations

### Risk 1 — game-introspector hallucinates state primitives
**Mitigation:** Inventory format includes file:line references for every claim. Validation step greps each claim before persistence. Sahil reviews first inventory per game before brain trusts it.

### Risk 2 — minimal hooks open security holes
**Mitigation:** All hooks gated by `import.meta.env.DEV` or equivalent build flag. Production builds must not ship hooks. hook-designer's PR template requires explicit gating + a smoke test asserting hooks are absent in production bundles.

### Risk 3 — Playwright drag-drop sequences for combo states are flaky
**Mitigation:** state-reacher prefers (a) hooks > (c) interaction sequences. Interaction sequences only used as fallback when no hook reachable AND game-side change refused.

### Risk 4 — brain becomes too autonomous, loses Sahil oversight
**Mitigation:** Every hook-designer output requires Sahil review before the builder subagent files the PR. Every new inventory requires Sahil review before brain trusts it. Brain produces proposals; Sahil ratifies.

### Risk 5 — scope creep into other agents
**Mitigation:** ASO brain charter explicitly lists out-of-scope items. Observations route to the right agent and continue. Single-purpose enforced by anti-pattern 6.
