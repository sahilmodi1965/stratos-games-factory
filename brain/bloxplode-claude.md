# CLAUDE.md

Conventions and rules for AI agents (Claude Code, the Stratos Games Factory daemon, subagents) and human contributors working in the Bloxplode repo. Humans should read `README.md` first.

## Project

**Bloxplode** — a combo-driven block puzzle game where every move can trigger a chain reaction (or end the run). Built as a hybrid mobile app via [Capacitor](https://capacitorjs.com/): the game itself is plain web (HTML/CSS/JS) under `www/`, wrapped into a native Android shell.

This repo is part of the [Stratos Games](https://github.com/sahilmodi1965/stratos-games-factory) portfolio. Play-test feedback comes in via GitHub Issues with the `build-request` label, and the Stratos daemon picks them up hourly. PR previews ship to `gh-pages /pr/<number>/`. The live web build is at `gh-pages /` (mirrored from `www/`).

## Repo layout

```
Bloxplode-Beta/
├── www/                     ← the actual game (HTML, CSS, JS, assets)
├── android/                 ← Capacitor-generated Android shell — DO NOT EDIT
├── capacitor.config.json    ← Capacitor config (app id, name, web dir)
├── package.json             ← Capacitor + plugin dependencies
└── .github/
    ├── workflows/           ← CI, PR previews, auto-merge, release, cleanup
    └── ISSUE_TEMPLATE/
        └── build-request.md ← used by play-testers to file work
```

The whole game lives in `www/`. That is where 99% of changes happen. The `android/` folder is generated by Capacitor (`npx cap sync`) and must never be hand-edited — any change there will be wiped on the next sync.

## Build / run

```bash
npm install                  # one-time
# To preview the game in a browser: open www/index.html (or use any static server in www/)
# To rebuild the Android app:
npx cap sync android         # copies www/ into android/, refreshes plugins
npx cap open android         # opens Android Studio for an APK build
```

There is no JS bundler. `www/` is shipped as-is. Keep imports plain (`<script src="...">`) and avoid build-step assumptions.

## Code conventions

- **Plain ES modules / vanilla JS / DOM / Canvas.** No frameworks (React/Vue/Svelte). No TypeScript.
- **No build step inside `www/`.** Files are shipped raw. If you need a helper, add another `.js` file and `<script>`-include it.
- **Keep files focused.** Aim for <200 lines per `.js` file when reasonable. Split by concern (rendering, state, input, audio) rather than by feature.
- **Asset paths are relative.** Capacitor serves `www/` as the document root, so `src="img/foo.png"` is the right form. Never use absolute paths starting with `/`.
- **Never reference Node APIs from `www/`.** It runs in a WebView, not Node.
- **Native plugin calls go through `@capacitor/...` imports** in the JS layer. Plugin config lives in `capacitor.config.json`.

## Things to never do

- **Never edit anything under `android/`.** It is generated. Hand-edits will be silently overwritten by `npx cap sync`. If you need a native change, edit the Capacitor config and re-sync — and flag it for a human reviewer.
- **Never `npm install` a new dependency without a clear reason in the issue.** Capacitor/AdMob/Crashlytics versions are pinned for a reason; bumping them can break the Android build.
- **Never modify `capacitor.config.json` autonomously.** App id, scheme, and plugin config changes require a human reviewer because they affect store identity and signing.
- **Never commit `node_modules/` or `android/build/`.**
- **Never add a bundler (Vite/Webpack/Rollup) to `www/`.** Keeping it raw HTML/JS is a deliberate design choice — it makes the game trivially debuggable in a browser.

## Stratos autobuilder rules (factory-managed — do not hand-edit)

This section is appended by the [Stratos Games Factory](https://github.com/sahilmodi1965/stratos-games-factory). It tells the daemon's headless `claude -p` session AND any direct human contributor how to operate in this repo. The factory owns everything between the BEGIN/END markers.

### When you are invoked by the Stratos daemon

You are running non-interactively under `claude -p`. You have one job: process a single GitHub issue (the daemon will tell you which one in the prompt) and either open a clean change set or refuse with a reason.

#### Hard rules (zero exceptions)

1. **Never edit `android/`.** It is generated by Capacitor. Hand-edits will be wiped on the next `npx cap sync`.
2. **Never edit `capacitor.config.json` autonomously.** App identity / plugin config changes require a human.
3. **Never bump dependency versions.** Capacitor/AdMob/Crashlytics versions are pinned. If the issue requires a bump, refuse and explain.
4. **All game changes go in `www/`.** That is the entire surface area you should be touching.
5. **Keep `www/` build-step-free.** Don't introduce a bundler, transpiler, or `package.json` script inside `www/`.

#### Scope discipline

- The issue describes ONE thing. Do that one thing.
- Do not refactor surrounding code. Do not "improve" naming. Do not add comments to code you didn't change.
- Do not add features that weren't requested.

#### Commits

- Conventional commits: `fix:`, `feat:`, `chore:`, `refactor:`, `style:`, `perf:`, `docs:`, `content:`, `level:`.
- One logical change per commit.
- Every commit message must reference the issue number (e.g. `fix: combo counter resets on game over #17`) so GitHub auto-links it.

#### When to refuse

Refuse (do nothing, leave the working tree clean, and explain in your final summary) if any of these are true:

- The issue is ambiguous and you would have to guess what "good" looks like.
- The fix requires touching `android/` or `capacitor.config.json`.
- The fix requires a dependency change.
- You cannot verify the change runs cleanly in a browser preview of `www/`.
- The issue is about native behavior (gestures, AdMob, store metadata) that needs an Android Studio rebuild.

Refusing is a successful outcome. The daemon will turn your explanation into an issue comment so a human can decide what to do next.

#### Final-step checklist before you stop

- [ ] Every commit references `#<issue-number>`.
- [ ] No edits inside `android/` or `capacitor.config.json`.
- [ ] No new or bumped dependencies.
- [ ] `www/index.html` opens cleanly (no console errors from your change).
- [ ] Your final response is a single paragraph summarizing the change (or the refusal).

### Direct contributor mode

Ripon and interns use their own Claude Code ($20 Pro plan) to push directly to this repo. **This is expected and encouraged** — the daemon and direct pushes coexist by design. Rules:

- **Push small changes** (content tweaks, asset updates, level data, copy edits, CSS tweaks, simple bug fixes inside `www/`) directly to `main`. Don't open a PR for a one-line fix.
- **Use feature branches** for anything that changes game mechanics, input handling, or core rendering. Open a PR for review.
- **Always pull before pushing**: `git pull --rebase origin main`.
- **Use conventional commits**: `fix:`, `feat:`, `chore:`, `content:`, `level:`. Reference an issue number when one exists.
- **Never delete `auto/*` branches manually.** The daemon owns them and the weekly cleanup workflow sweeps them.
- **If CI fails after your push, fix it immediately or revert**: `git revert HEAD && git push`. Don't leave `main` red.
- **Auto-merged PRs** ship to the web preview instantly. The Android APK still requires a manual rebuild on Sahil's Mac (`npx cap sync android` → Android Studio).

### Priority of work (for play-testers)

If you are a human play-tester (Ripon or an intern), your job is **NOT to write code**. Your job is, in priority order:

1. **Play the game obsessively.** Find every tiny thing — bugs, awkward feels, missing polish, combo timing, sound, animation pacing.
2. **File detailed issues** for everything. Use the `Build Request` template. One thing per issue. The daemon picks them up hourly.
3. **Use your own Claude Code ($20 Pro) for quick 2-minute fixes** if you want — color tweaks, copy edits, level number changes. Push directly to main.
4. **Test every PR preview.** When the daemon comments with a preview URL, click it (on your phone, in mobile mode), play the change, comment your verdict.
5. **Test every auto-merged change** on the live URL within an hour of the merge.
6. **Add levels and content** — as many as you can. Adding a level data file or content JSON is auto-mergeable.
7. **When the game feels ready for a release**, talk to Sahil and add the `ship-it` label to a tracking issue or PR. That triggers the release workflow (web). The Android APK rebuild still needs Sahil's Mac.

What you should NOT spend time on:
- Editing `android/` or `capacitor.config.json` (off-limits).
- Bumping Capacitor / plugin versions (off-limits).
- Refactoring core game logic in `www/*.js` files without a clear bug.
