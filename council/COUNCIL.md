# Stratos Factory Council

This document is the **factory's living memory**. The daemon reads it at the
start of every build run and appends it to Claude's system prompt, so any
guidance written here flows directly into every build attempt.

The weekly `council/review.sh` script (cron: Sunday 00:00 UTC) appends to
this file via Claude itself, based on what actually broke or worked in the
past 7 days. The council replaces hard-coded validation policy with a
living, data-driven feedback loop: instead of someone prescribing what
to check, the council figures out what to check based on what actually
breaks.

## Format

Each entry is one of four types:

- `### Lesson learned` — a postmortem fact derived from a real incident
- `### Known issue` — something currently broken or partly broken
- `### Architecture decision` — a design choice + the reason it was made
- `### Improvement suggestion` — a recommendation that hasn't been implemented yet

Every entry must be:
- **Concrete** — cite a real failure mode, PR number, build.log timestamp,
  or commit hash
- **Actionable** — Claude (or the next reviewer) can apply it to the next
  build
- **Time-stamped** — `(YYYY-MM-DD)` so we know how stale it is

The weekly review script appends, never rewrites. History matters.

## Council guidance for Claude (when invoked by the daemon)

When you read this file as part of your system prompt, you are not asked to
implement anything in it. You are asked to **let it shape your next build**:

1. The "Known issue" entries tell you about parts of the system that are
   currently broken or unreliable. Avoid those landmines.
2. The "Lesson learned" entries are hard-won insights from previous runs
   that went wrong. Apply them.
3. The "Architecture decision" entries explain why the system is the way
   it is. Do not undo a decision unless the issue you're working on
   explicitly asks you to revisit it.
4. The "Improvement suggestion" entries are NOT for you to act on. They
   are tracked separately by the human team.

# Seeded entries (2026-04-10 — initial population from the original build session)

### Lesson learned — daemon must scrub build output before commit (2026-04-10)

The first generation of the daemon ran `git add -A` after Claude finished,
which swept Vite's `docs/` build output into every PR. The "broken" PRs
#17/#18/#19 were poisoned by the daemon committing its own build output, NOT
by Claude editing forbidden paths. Fix in place: `daemon/stratos-daemon.sh`
now resets the per-game forbidden_paths field (`docs:packages:prototypes`
for arrow-puzzle, `android:capacitor.config.json:package-lock.json` for
bloxplode) to HEAD after Claude's session, before sweeping any trailing
changes.

**Council guidance for Claude**: do NOT trust `git add -A`. Use targeted
`git add <named-files>` only. The daemon's safety net will catch you if
you forget, but explicit is better.

### Lesson learned — system prompt > user message for sticky rules (2026-04-10)

The first daemon prompt put hard rules in the user message. After many tool
turns, Claude de-prioritized them and started touching forbidden paths.
Moving the rules to `--append-system-prompt` (loaded from
`templates/system-prompt-<game>.md`) made them sticky across every turn
and fixed the issue.

**Council guidance for Claude**: when you read this entry, that is your
reminder to re-check the hard rules in your system prompt before any write
operation. Especially before any `git add`.

### Lesson learned — issue premises often don't match the codebase (2026-04-10)

Issue #15 ("Speed round: 4x4 level with 30-second timer") said "Level
declares `timer: 30` in its JSON" — but Arrow Puzzle has NO per-level JSON
files. Levels are 100% procedurally generated by `level-loader.js` from
`difficulty-config.js`. The first three broken PRs (#17/#18/#19) all tried
to invent JSON-level systems that didn't exist. The PR that actually worked
(#21, after the daemon rewrite) added `SPEED_ROUNDS` to `difficulty-config.js`
in the existing pattern instead of inventing a new format.

**Council guidance for Claude**: before implementing ANY level/feature
work, verify the issue's implementation suggestion against what actually
exists. Inventing parallel systems is never correct. Match the existing
pattern or refuse with an explanation.

### Lesson learned — PR previews need relative asset paths (2026-04-10)

The PR-preview workflow deploys each PR build to `gh-pages/pr/<num>/`.
`vite.config.js` was originally configured with `base: '/arrow-puzzle-testing/'`
(absolute), so the built `index.html` referenced assets at the root. At a
preview URL like `.../arrow-puzzle-testing/pr/21/`, the browser fetched
`/arrow-puzzle-testing/assets/...` (root) instead of
`/arrow-puzzle-testing/pr/21/assets/...` (subpath) → 404 → unstyled HTML.
Fix in place: `vite.config.js` now uses `base: './'`. Both the live site
and any subpath preview now resolve correctly.

**Council guidance for Claude**: never override `base` in `vite.config.js`
without understanding both the live-site path AND the preview-path
implications. If you need to change it, document why in this file.

### Lesson learned — workflow YAML literal-block bash strings (2026-04-10)

`auto-merge.yml` and `release.yml` had a hidden YAML parse bug from day
one: bash strings inside `run: |` blocks that spanned literal newlines
without proper indentation, which made YAML think the literal block had
ended. Workflows registered with the file path as their name and never
fired their real triggers. The early "validation pass" was a false
positive — PyYAML wasn't actually installed in the env that ran the
check, and the loop's exception handler silently exited 0.

**Council guidance for Claude**: when authoring or editing GitHub Actions
YAML, validate it with a real YAML parser AND verify GitHub recognizes the
workflow name (not the file path) in `gh workflow list`. If the run name
shows the file path, the file failed parse — fix it before assuming it
works.

### Lesson learned — bash 3.2 + cron PATH on macOS (2026-04-10)

macOS ships bash 3.2 by default and cron has a minimal PATH that excludes
Homebrew, nvm, and `~/.local/bin`. Two early failures:

- `gh: command not found` at the 02:00 cron tick (fixed: `export PATH=...`
  at top of `daemon/stratos-daemon.sh`)
- `seq 0 -1` produces "0\n-1" on BSD seq instead of nothing on GNU seq
  (fixed: replaced with `while ((++i < n))` loop)

**Council guidance for Claude**: any new shell code in the daemon must be
tested under `/bin/bash` (3.2) and BSD coreutils, not GNU. Use `awk` for
text processing — it's portable. Avoid `sed -i ''` vs `sed -i` divergence
by using `awk` + tmp file instead.

### Architecture decision — structural validators only, until council recommends more (2026-04-10)

The factory currently runs `npm run validate` (paths + difficulty schema)
as its only post-Claude gate. It does NOT yet run the game in a headless
browser, take screenshots, or replay gameplay. This is deliberate: before
adding more validation infrastructure, let the council collect a few weeks
of real failure data and recommend the cheapest validators that would catch
the most failures.

Implementations explicitly **deferred** until the council recommends them
based on data:

- `window.render_game_to_text()` — agent-readable game state snapshot
- Headless puppeteer/playwright runtime QA
- Screenshot/vision QA via a second Claude session
- Action-replay-based gameplay validation
- 5-phase QA gate with bounded retries inside Claude's session

The council should look at what actually broke in the past week and
recommend which (if any) of these are worth building.

### Architecture decision — daemon prompt structure: system + compact user (2026-04-10)

Claude's session is invoked with:
- `claude -p --dangerously-skip-permissions --effort max`
- `--append-system-prompt <full content of templates/system-prompt-<game>.md
   PLUS the contents of council/COUNCIL.md>`
- A short user message containing only the issue body and a numbered to-do

The system prompt carries the immutable rules and the council's living
memory. The user message just delivers the per-issue context. This
separation is what made the rules stick across many tool turns.

**Council guidance**: do NOT inflate the user message with rules. Add new
guidance here (in COUNCIL.md) or to the per-game system prompt template,
never to the user message.

### Known issue — dedup heuristic closed issue #16 as a false positive (2026-04-10)

The recent-commit dedup (see `recently_addressed` in
`daemon/stratos-daemon.sh:66-103`) closed **issue #16** ("Document
difficulty curriculum in Arrow_Puzzle_Game_Plan.md") with the
auto-comment *"looks like this was already addressed in commits within
the last 24 hours"* at 2026-04-09T20:56:27Z. The claim is wrong.

Evidence:
- `git grep -l "Difficulty Curriculum"` in `arrow-puzzle-testing` returns
  nothing. The section the issue asked for was never added.
- The only commit in the 24h window that touched
  `Arrow_Puzzle_Game_Plan.md` was `3f5546b` ("fix(arrow-puzzle): play
  button now switches to game screen + docs cleanup") and it changed 5
  lines of cleanup, not the curriculum section.
- Why the heuristic fired: the issue title decomposes into significant
  words `document, difficulty, curriculum, arrow, puzzle, game, plan`.
  Commit `3f5546b` contains `arrow, puzzle, play, button, switches,
  game, screen, docs, cleanup`. The overlap is `arrow + puzzle + game`
  = exactly 3 — the threshold. **The repo-name noise alone ("arrow",
  "puzzle", "game") hits the threshold** before any real semantic match
  happens.

This confirms the seeded 2026-04-10 improvement suggestion worry:
*"Decide whether the recent-commit dedup heuristic (3+ keyword overlap)
is producing false positives in real data."* Yes it is, on the very
first week of real data.

**Council guidance for Claude**: when editing the daemon's dedup logic,
do not raise the threshold (that would silence legitimate dupes). Build
a repo-specific stop-word list that strips the repo name, the game
name, and any word appearing in >=50% of recent commit subjects before
counting overlap. Filed as a council issue with the proposed fix.

### Lesson learned — 03:00:00 cron tick failed with "gh CLI not authenticated" (2026-04-10)

`build.log` shows two back-to-back `[2026-04-10 03:00:00] FATAL: gh CLI
not authenticated. Run 'gh auth login'.` lines. The 02:25:01 run 35
minutes earlier (manual, with `REPO_FILTER`/`ISSUE_FILTER` set) passed
the identical preflight (`daemon/stratos-daemon.sh:114`) and built PR
#21 to completion.

Confirmed in-session that interactive `gh auth status` still succeeds
under `sahilmodi1965 (keyring)` with `gho_…` token scopes
`gist, read:org, repo, workflow`. The failure is therefore not "token
expired" — it's environment-specific. Most likely cause on macOS: the
gh token is stored via the keyring credential helper, and cron runs in
a context where the login keychain isn't unlocked, so `gh auth status`
cannot read the token back.

This means **every hourly cron tick on Sahil's machine is at risk of
silently failing the preflight** until the auth storage is migrated or
the daemon learns to self-heal. The only reason the 02:25:01 run
succeeded is that it was a manual invocation inheriting the interactive
shell's keychain access.

**Council guidance for Claude**: do not edit the daemon preflight to
"skip on auth failure" — that would turn a loud failure into a silent
one. The right fix is diagnosing why the keychain is unreachable from
cron and either (a) moving to a file-based token, or (b) moving the
daemon to launchd. Filed as a council issue.

### Improvement suggestion — capture per-run metrics to `council/runs.jsonl` (2026-04-10)

The weekly review script only has access to `build.log` (unstructured
text) and `gh` API queries. It cannot directly compute merge rate,
validator failure rate, average build time, or failure-mode histogram.
This is why the first weekly review has to reason from `grep` over
free-form log lines.

Proposal: have `daemon/stratos-daemon.sh` append one JSONL record per
processed issue to `council/runs.jsonl`, with fields:

```
{"ts":"2026-04-10T02:36:23Z","repo":"mody-sahariar1/arrow-puzzle-testing",
 "issue":15,"branch":"auto/...","claude_exit":0,"validate_exit":0,
 "rebase_clean":true,"changed_files":6,"pr":21,"outcome":"pr_opened"}
```

With that, the next weekly review can answer: "what % of daemon runs
produced a merged PR?", "which validators fire on which issues?",
"how long does claude take?", without re-parsing text logs. This is a
prerequisite for choosing the "cheapest validator that catches the most
failures" — we cannot rank failure modes we haven't counted. Filed as
a council issue.
