# Forbidden paths — what they are, why they exist, what breaks if we open them

**Audience:** Ripon (and any future contributor who asks "why can't the factory touch this?")
**Last updated:** 2026-04-20

---

## 🧒 Explain-like-5

🏭 **The factory is a robot** that builds game changes. It's very fast and careful, but it has **no hands for some kinds of tools**.

🔧 **Some files need hands the robot doesn't have** — like:
- A **physical phone** to test Android builds (`android/`, `ios/`, `capacitor.config.json`)
- A **whole-portfolio brain** to know that changing shared code breaks other games (`packages/`)
- A **memory of the original design** to compare against (`prototypes/`)

🚪 **Forbidden paths are rooms the robot isn't allowed to enter.** If it walked into the "Android build room" and moved things around, no one could check if the room still works until you (Ripon) got there — and by then the robot would have built ten more things on top of the broken one.

🧸 **Analogy:** It's like a very smart toddler who cleans the kitchen perfectly. You don't let them handle the knife drawer, not because they'd cut themselves, but because **only you know where each knife goes back**. The factory handles web code great; only you can handle phone builds, shared kits, secrets, and lockfiles.

💡 **Why it matters:** Without forbidden paths, the robot would "fix" things it can't actually test, ship green CI builds that crash on real phones, and silently break games you aren't even reviewing that week. The rule is **slower on purpose** so that what ships is what was tested.

## What each forbidden path guards against — and the risk if opened

### `docs/` / `dist/` / `node_modules/` — build output

**Guards against:** source ≠ deployed output drift.

**Risk if opened:**
- Factory commits stale build output → GitHub Pages deploys yesterday's code while source shows today's fix
- Ripon reviews the PR source (looks correct) → site still shows the bug → false failure → hours lost debugging a diff that isn't real
- `node_modules/` committed = repo bloats from ~1MB to ~500MB, git operations crawl

**Why the rule works:** CI rebuilds on every merge. The only canonical output lives on Pages, rebuilt from source. Source is the contract.

---

### `packages/` — shared kits

**Guards against:** a fix in one game silently breaking another.

**Risk if opened:**
- Factory edits `@audio` for arrow-puzzle's haptics → change ships → Bloxplode's rumble-on-explosion regresses
- Ripon reviews arrow-puzzle, doesn't open Bloxplode that week → Bloxplode ships the regression to Play Store
- Every shared kit change needs a three-game regression check, which only a human with the whole portfolio in mind can do

**Why the rule works:** kit changes require an explicit human-authored "kit PR" on the factory repo, with sign-off from all three game owners, not a game-PR side-effect.

---

### `prototypes/` — frozen reference

**Guards against:** losing the "before" for future refactors.

**Risk if opened:**
- Factory "tidies" the original tutorial code → comparison of "how did this used to work?" becomes impossible
- When a new engineer asks "why was the arrow-angle formula like this in the old code?" the answer is gone
- Research artifact turns into production-drift

**Why the rule works:** prototypes/ is the git-tracked museum. Look, don't touch.

---

### `android/` / `ios/` / `capacitor.config.json` — native mobile

**Guards against:** shipping a "fix" the factory can't test.

**Risk if opened:**
- Factory "fixes" the Gradle Kotlin stdlib issue (#172) → commit goes in → CI is web-only, so it passes green → Ripon installs the APK → crash on launch
- Factory bumps a native plugin version → APK signs but segfaults on a specific Android version → no way to detect without real device matrix
- Factory "helpfully" adds an Info.plist permission → App Store rejection on next submission → release delayed

**Why the rule works:** everything native goes through Ripon's device + `./gradlew assembleDebug` / `npx cap run ios` on his machine. The green-CI lie the web layer can tell you does not translate to native.

**Current live example:** arrow-puzzle PR #182 carries a 🚫 BLOCKED-by-#172 banner. #172 wants a 3-line `android/app/build.gradle` edit to resolve a Kotlin stdlib duplicate-class. That edit is blocked for the factory; Ripon will apply + verify the fix on his device, merge #172, then PR #182 unblocks. **This is the rule working as designed.**

---

### `package-lock.json` — dependency lock

**Guards against:** phantom "works on CI / broken on your machine" divergences.

**Risk if opened:**
- Factory bumps `@capacitor/haptics` from 7.2.0 to 7.3.0 to "stay current" → CI installs 7.3.0 → Ripon's last `npm install` was 6 weeks ago and his cache has 7.2.0 → "I can reproduce your bug" becomes "wait, I can't"
- Silent dep bumps hide breaking changes inside patches (semver is advisory, not enforced)
- Supply-chain risk: an upstream dep compromise lands in CI before a human even reviewed the bump

**Why the rule works:** dep bumps are an explicit, human-authored PR class with attached release-notes review. Never a side-effect of a feature PR.

---

### Secrets — values, never files

**Guards against:** credential leak.

**Risk if opened:**
- Factory commits a Firebase API key into source → key is in git history forever → public repo → key harvested by scrapers → $50k billing event in a week
- AdMob app ID leaked → competitors spam fake impressions → account flagged → app pulled
- Signing-cert password in CI logs → anyone reading public workflow logs can re-sign and impersonate the app
- Play Store integrity check fails on next submission → submission blocked, sometimes account-level

**Why the rule works:**
- Factory code writes `process.env.FIREBASE_API_KEY` (the **reference**)
- `.github/workflows/*.yml` writes `${{ secrets.FIREBASE_API_KEY }}` (the **reference**)
- Ripon runs `gh secret set FIREBASE_API_KEY` once per game (the **value**, never touches git)
- The factory's brain never sees or handles the value. Full spec: `council/SECRETS.md`.

## What's *not* forbidden (common confusions)

- **`.github/workflows/*.yml`** — CI definitions. Factory CAN edit these (adding runtime-smoke, validators, etc.). They reference secrets via `${{ secrets.X }}` which is fine.
- **`games/<game>/src/**`** — all game code. This is the factory's primary work area.
- **`tests/`** — unit tests, smoke tests, playwright specs. Factory expects to write these.
- **`vite.config.js`** / **`playwright.config.js`** / **`package.json`** (scripts section) — build config. Factory edits as needed; only `package-lock.json` is frozen.
- **`CLAUDE.md`** on game repos — game-level brain, factory CAN update these to encode per-game rules.
- **`council/` / `templates/` / `scripts/`** on the factory repo — the factory-level brain, edited all the time.

## The meta-rule

**Forbidden paths are the shape of "what the factory can't verify."** Each line of the list traces back to a real verification gap:

- `docs/` — factory can't verify what Pages actually serves
- `packages/` — factory can't verify cross-game regression
- `prototypes/` — factory can't verify "was this intentional?"
- `android/` / `ios/` — factory can't verify native install + runtime
- `package-lock.json` — factory can't verify reproducibility across machines
- Secrets — factory must not handle values it can't rotate

**If a path gets added to this list in the future, it's because someone caught a new verification gap.** If a path gets removed, it's because we've built a validator that closes the gap (e.g. a fully-device-sandboxed CI matrix would retire the `android/` rule).

The current list reflects **F1 reality (2026-04-20):** we are shipping one real game end-to-end, and the verification infrastructure is Ripon's device + his Claude Code session + CI's web-only build. Expand the list when a new gap is found; shrink it when a validator makes the gap impossible.
